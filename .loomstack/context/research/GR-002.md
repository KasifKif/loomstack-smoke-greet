Pytest, a widely-used Python testing framework, relies on specific patterns for test discovery and organization, promoting clear, maintainable, and scalable test suites. These patterns encompass naming conventions, file structures, and advanced features like fixtures and parametrization.

Here are the common pytest test patterns:

### 1. Test Discovery Naming Conventions

Pytest automatically discovers tests based on predefined naming conventions:
*   **Test Files**: Test files should be named `test_*.py` or `*_test.py`. Pytest searches the current directory and its subdirectories for files matching these patterns.
*   **Test Functions and Methods**: Test functions and methods must start with the prefix `test_` (e.g., `test_example_function()`). Function names should be descriptive and follow the `snake_case` convention.
*   **Test Classes**: Test classes must start with `Test` (e.g., `TestUserAuthentication`) and should not have an `__init__` method. Test methods within these classes still need to begin with `test_`.
*   **Assertions**: Pytest uses Python's built-in `assert` statements for checking outcomes, making tests more concise and Pythonic.

### 2. Test File and Directory Structure

While pytest doesn't enforce strict directory naming, it's recommended to organize tests logically:
*   **Separate Test Directories**: It's a common practice to keep test directories separate from application code, often under a `tests` folder in the project root.
*   **No `__init__.py` in Test Directories**: Generally, avoid placing `__init__.py` files in your primary test directories, as it's not often necessary to make the test directory importable and can sometimes confuse package discovery. However, for "inlined tests" within an application package, `__init__.py` files might be present. If you need to have test modules with the same name, adding `__init__.py` files to your test folders can turn them into packages, allowing pytest to load them correctly.
*   **`conftest.py` for Sharing Fixtures and Configuration**: `conftest.py` files are used to share fixtures across multiple test files in a directory or its subdirectories without explicit imports. They can also hold pytest hooks and configuration settings.

### 3. Fixtures

Pytest fixtures are essential for setting up and tearing down test environments, providing consistent conditions for tests.
*   **Definition**: Fixtures are defined using the `@pytest.fixture` decorator.
*   **Purpose**: They help generate and initialize test data, fake objects, application state, or configuration. Fixtures promote modularity by abstracting setup/teardown logic into reusable components.
*   **Scope**: Fixtures can have different scopes (e.g., `function`, `class`, `module`, `session`) controlling how frequently they are set up and torn down.
*   **Factory Fixtures**: For creating multiple variations of objects for tests, the factory pattern can be used with fixtures to ensure flexibility and maintainability.

### 4. Parametrization

Parametrization allows running a single test function with multiple sets of arguments, reducing redundancy and improving readability.
*   **`@pytest.mark.parametrize`**: This marker is used to define multiple sets of parameters directly within the test function, making it easier to manage and scale tests.
*   **Fixture Parametrization**: Fixtures can also be parametrized to return different values for each test run, allowing the same functionality to be tested across various inputs without duplicating test code.

### 5. Customizing Test Discovery and Behavior

Pytest offers ways to customize its default test discovery and execution:
*   **Configuration Files**: You can configure pytest's behavior, including changing naming conventions for files, classes, and functions, by setting options like `python_files`, `python_classes`, and `python_functions` in a configuration file (e.g., `pytest.ini`, `pyproject.toml`, `setup.cfg`).
*   **Ignoring Paths**: Specific directories or modules can be ignored during test collection using the `--ignore` or `--ignore-glob` command-line options or by setting `norecursedirs` in a configuration file.
*   **Markers**: Tests can be marked with custom attributes using `@pytest.mark.<name>` to categorize them, allowing for selective execution or special handling.

### 6. Other Useful Patterns

*   **Temporary Directories/Files**: Pytest provides a built-in `tmp_path` fixture to request a unique temporary directory for functional tests, which is automatically managed.
*   **Monkeypatching/Mocking**: Pytest includes features for monkeypatching and mocking modules and environments, which is useful for isolating code under test and controlling external dependencies.
*   **Integration with `unittest.TestCase`**: Pytest can discover and run tests written using the standard `unittest.TestCase` subclassing technique.
